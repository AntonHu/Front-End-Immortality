# 写技术实践博客

## 使用时机

- 准备对外分享一套架构、实践经验（如 AI Coding、规范体系、工作流），需要写成独立、可阅读性强的博客时。
- 已有一份「内部设计文档 / 架构说明」，需要改写成对外博客（去掉项目名、内部路径，补足引言与案例）时。
- 写完初稿后，想系统检查：是否有读者视角、是否有亮点总结、是否有足够但不过量的代码片段、是否需要拆分为多篇系列时。

## 核心原则

- **读者视角优先**：先问读者痛点，不要从「我要介绍一套架构」开头，而是从「你是不是经常遇到 X 问题」切入，再引出这套方案能解决什么。
- **亮点先说清楚**：在开头 1～2 段，用一小节「亮点」说明这套方案最厉害的 2～3 件事（例如：页面 90% 还原设计稿、设计稿信息直达代码、Rules/Skills 渐进披露等）。
- **标题短、无序号**：章节标题只写核心名词（如「SDD」「MCP」「Rules」「Skills」「完整流程」），不带「1. 什么是…」之类前缀；避免在标题中写数字序号。
- **先概念，立即跟「怎么用」**：每个概念性章节控制在几行内，立刻补上「在这套架构里怎么用」「目录大概长什么样」「什么时候用到」。
- **用「最小可落地骨架」，不复制完整实现**：示例以「YAML 头 + 几个小节骨架 + 简短代码片段」为主，避免整篇贴满完整文件；完整实现放仓库，博客只给读者能照着抄的框架。
- **渐进披露**：长内容拆成多篇或多层次：首篇讲整体架构与关键路径，后续篇分别讲 Rules、Skills 等细节，并相互跳转。
- **内容与项目路径解耦**：博客中**不出现**读者看不到的内部路径（如 `.agents/rules/01-项目概述.md`），要么贴出简化内容，要么用通用描述 + 代码片段代替。
- **语气自然，避免「我要写一篇博客」这种元叙述**：直接以作者口吻开讲经验，不要在文中强调「下面我会用博客形式」「这篇文章将会怎样怎样」。

## 写作流程（6 步）

1. **收集素材与目标**  
   - 梳理：这篇要解决读者的什么痛点？你这套实践最核心的 2～3 个亮点是什么？  
   - 收集：相关的代码片段、目录结构、规则/技能示例、流程图（如 mermaid）。

2. **设计整体结构**  
   - 通用骨架（可按需调整）：  
     1）引言：痛点 + 亮点  
     2）架构/主题概览：用一张表或一段话说清组件（SDD/MCP/Rules/Skills…）及关系  
     3）关键组成章节：每个模块一小节（例如 SDD、MCP、Rules、Skills）  
     4）完整流程或实战示例（最好配一张 mermaid 流程图）  
     5）总结与展望，指向仓库或续作。  
   - 若内容天然很大（如 Rules、Skills），在结构设计阶段就预留「续作篇」：首篇只讲历程和清单，细节放后续。

3. **撰写开头：痛点 + 亮点**  
   - 用 1～2 段提出典型场景/痛点：如「AI 生成代码风格不一致」「设计稿还原度低」「返工频繁」等。  
   - 紧接着一小段「亮点」：用加粗列出这套实践带来的关键收益（如「页面 90% 还原设计稿」「一套 Rules/Skills 多 Agent 共用」「SDD 让需求/变更有记录可回溯」）。

4. **按模块写正文：先说是什么，再说我怎么用**  
   - 对每个模块（例如 SDD、MCP、Rules、Skills）：  
     - 用 2～3 句话解释「是什么、解决什么问题」；  
     - 紧接一个小节讲「我在这个项目里怎么用」，包括：  
       - 目录大概长什么样（用简化 text 目录或表格）；  
       - 什么时候会用到它（流程中的环节）；  
       - 用 1～2 段示例代码/配置（如 YAML 头、规则/技能骨架），避免贴满整文件。  
   - 避免长篇抽象讲历史或理论，只保留与实践直接相关的说明。

5. **处理篇幅与系列划分**  
   - 当发现单篇过长、且某一部分（如 Rules、Skills）可以作为独立话题讲得更细时：  
     - 在首篇中保留**演进历程 + 设计原则 + 清单**这类「鸟瞰」内容；  
     - 把大量示例、演进细节、对比、更多代码片段拆到续作里；  
     - 在首篇对应位置加一句简短引导：「详细实践见《XX 篇》」。

6. **输出前自查清单**  
   - **结构**：  
     - [ ] 是否有清晰的引言（痛点 + 亮点）？  
     - [ ] 是否有一段/一表「整体架构/主题概览」？  
     - [ ] 是否有一节「完整流程 / 实战链路」（最好用 mermaid）？  
     - [ ] 是否需要拆分为系列，并互相有简短跳转说明？  
   - **内容密度**：  
     - [ ] 每个概念性章节是否在几行内说完「是什么 + 解决什么问题」？  
     - [ ] 是否立刻跟上「怎么用」和「目录/代码骨架」？  
     - [ ] 示例是否为「最小可落地骨架」，而不是完整大文件？  
   - **风格与读者体验**：  
     - [ ] 标题是否短、无显式序号（1. 2. 3.）？  
     - [ ] 是否避免在文中说「我会用博客形式」「这篇博客将…」，而是直接进入内容？  
     - [ ] 是否去掉了内部私有路径，改为内容本身或通用路径示例？  
     - [ ] 是否有足够的加粗小标题/表格，方便扫读和查找？  
   - **落地性**：  
     - [ ] 读者按文中的目录/代码骨架，能否在自己项目里搭出类似结构？  
     - [ ] 是否给出必要的「命令入口」或「操作入口」（如 `openspec init/proposal/apply/archive`、如何配置 `.agents`、如何挂 Skills/Rules）？

## 如何在本项目中使用

- 当你准备写类似「前端 AI Coding 落地指南」这种文章时，可以先用本 SKILL 的流程梳理结构与素材，再动手写正文。
- 若后续你在其他项目中也有对外分享，只需复用这套骨架：替换架构内容、示例和流程图即可，写作思路保持一致。
