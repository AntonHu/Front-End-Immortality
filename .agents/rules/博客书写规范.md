# 博客书写规范

本规范适用于面向技术社区的实践型博客，与 `.agents/skills/write-blog` 配套使用：Skills 提供流程与示例，本文档约定写作原则与输出前自查标准。

---

## 核心原则

- **读者视角优先**：先问读者痛点，不要从「我要介绍一套架构」开头，而是从「你是不是经常遇到 X 问题」切入，再引出这套方案能解决什么。
- **亮点先说清楚**：在开头 1～2 段，用一小节「亮点」说明这套方案最厉害的 2～3 件事（例如：页面 90% 还原设计稿、设计稿信息直达代码、Rules/Skills 渐进披露等）。
- **标题短、无序号**：章节标题只写核心名词（如「SDD」「MCP」「Rules」「Skills」「完整流程」），不带「1. 什么是…」之类前缀；避免在标题中写数字序号。
- **先概念，立即跟「怎么用」**：每个概念性章节控制在几行内，立刻补上「在这套架构里怎么用」「目录大概长什么样」「什么时候用到」。
- **用「最小可落地骨架」，不复制完整实现**：示例以「YAML 头 + 几个小节骨架 + 简短代码片段」为主，避免整篇贴满完整文件；完整实现放仓库，博客只给读者能照着抄的框架。
- **渐进披露**：长内容拆成多篇或多层次：首篇讲整体架构与关键路径，后续篇分别讲 Rules、Skills 等细节，并相互跳转。
- **内容与项目路径解耦**：博客中**不出现**读者看不到的内部路径（如 `.agents/rules/01-项目概述.md`），要么贴出简化内容，要么用通用描述 + 代码片段代替。
- **语气自然，避免「我要写一篇博客」这种元叙述**：直接以作者口吻开讲经验，不要在文中强调「下面我会用博客形式」「这篇文章将会怎样怎样」。

---

## 输出前自查清单

- **结构**
  - [ ] 是否有清晰的引言（痛点 + 亮点）？
  - [ ] 是否有一段/一表「整体架构/主题概览」？
  - [ ] 是否有一节「完整流程 / 实战链路」（最好用 mermaid）？
  - [ ] 是否需要拆分为系列，并互相有简短跳转说明？

- **内容密度**
  - [ ] 每个概念性章节是否在几行内说完「是什么 + 解决什么问题」？
  - [ ] 是否立刻跟上「怎么用」和「目录/代码骨架」？
  - [ ] 示例是否为「最小可落地骨架」，而不是完整大文件？

- **风格与读者体验**
  - [ ] 标题是否短、无显式序号（1. 2. 3.）？
  - [ ] 是否避免在文中说「我会用博客形式」「这篇博客将…」，而是直接进入内容？
  - [ ] 是否去掉了内部私有路径，改为内容本身或通用路径示例？
  - [ ] 是否有足够的加粗小标题/表格，方便扫读和查找？

- **落地性**
  - [ ] 读者按文中的目录/代码骨架，能否在自己项目里搭出类似结构？
  - [ ] 是否给出必要的「命令入口」或「操作入口」（如 `openspec init/proposal/apply/archive`、如何配置 `.agents`、如何挂 Skills/Rules）？
